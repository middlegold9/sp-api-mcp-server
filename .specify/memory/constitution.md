# Constitution — sp-api-mcp-server

> Spec-Kit 项目治理原则。任何代码前先有 spec → plan → tasks。

## 核心原则（FDE 方法论对齐）
1. **规格驱动**：先 `spec.md`（做什么/为什么）→ `plan.md`（怎么做）→ `tasks.md`（按序实现）。禁止无规格写码。
2. **高速公路地基**：本服务是上层 Agent 的统一数据/执行层；只封装"对接亚马逊"，不写业务策略（调价/选品逻辑在上层）。
3. **人类在环**：所有写工具（feed 提交、ads 更新）必须经审批网关；默认拒绝，需显式开启 + 人工确认。
4. **数据隐私优先**：PII 与报表默认本地；外传需显式开启。
5. **最小权限**：SP-API / 广告 API 仅申请必需 scope；RDT 仅在必要时换取，dataElements 精确声明。
6. **不踩政策红线**：亚马逊政策硬约束（如不能代为索评）；护栏拦截。
7. **可验证**：每个功能有量化验收（eval/测试），禁止"看起来能用"。
8. **可复盘**：结构化日志 + 指标 + 工具版本化，成果回流平台（碎石路→高速公路）。
9. **YAGNI / DRY**：只写当前需要的代码，不提前抽象。
10. **TDD**：实现前先写失败测试（RED→GREEN→REFACTOR）。
11. **工作隔离**：用 git worktree 并行开发，干净基线再动手。

## 技术约束
- 语言：Python 3.11+；框架：FastMCP (`mcp`)；传输：stdio（默认）/ SSE（可选）。
- 默认分支：`main`；提交信息用英文、祈使句。
- 密钥绝不入库；提供 `.env.example`。
